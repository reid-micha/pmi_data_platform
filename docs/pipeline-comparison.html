設計決策 D1–D9 Design decisions

  • D1 Typed multigraph — 邊帶 type / weight / direction / confidence，不是單一 scalar。市場關係是有語意的圖，不是純統計距離矩陣。 Edges carry type, weight, direction, confidence — not a single scalar. Market relationships are semantically typed, not just statistical distances.
  • D2 結構邊 vs 統計邊分流 — core_market_edges 存靜態語意關係（agent 判定、進 audit）；ts_market_correlations 存動態統計相關（漂移、定期重算）。兩者不混用。 Structural edges (agent-classified, audit-grade, in core_market_edges) and statistical edges (drift-prone, periodically recomputed, in ts_market_correlations) are stored and managed separately.
  • D3 三量分清 — 機率 = 被加權的值 (0..1)；volume = 權重 mass（正交）；佔比 share = family 內 normalize 的衍生視角。三者不互換，命名 / schema 欄位必須區分。 probability, volume (weight mass), and share (derived normalization view) are distinct concepts; naming and schema columns must never conflate them.
  • D4 agent 提議邊不歸類節點 — edge-proposer 判的是兩市場的關係，不是單一市場的類別。節點分類由 selector / index def 負責，不下放給 edge agent。 The edge-proposer agent classifies relationships between market pairs, not individual market categories. Node classification remains the selector's and index def's responsibility.
  • D5 Formula registry，不做 eval() — formula 字串不能任意執行（RCE 雷）。新 formula 要寫 Python，過 code review，加入 registry 後才可用。 Formula dispatch uses a named registry, never eval() of arbitrary strings (RCE risk). New formulas require code review and explicit registration.
  • D6 留 Postgres，不上 Neo4j — recursive CTE + in-memory networkx 足以應付 P0/P1 圖規模（<100k 邊）。等圖規模真的撐不住才考慮圖 DB。 Recursive CTEs + in-memory networkx handle P0/P1 graph scale (<100k edges). Graph DB only if Postgres genuinely can't keep up.
  • D7 邊全域共用、雙迴圈解耦 — 昂貴的 LLM 邊判斷（Loop A）攤平到所有 index 分攤，不在每個 index 的 scoring tick 重跑。Loop B 只讀邊，不寫邊。 Expensive LLM edge classification (Loop A) is amortized globally across all indexes. Loop B is read-only with respect to edges, keeping scoring ticks cheap.
  • D8 Graceful degradation — 邊表空或 agent 服務不可用時，Loop B 退化成現況扁平行為，不報錯，不中斷 scoring。 When the edge table is empty or the agent service is unavailable, Loop B degrades to current flat-list behavior without error or scoring interruption.
  • D9 GraphRAG 借 pattern，不 adopt framework — 從 GraphRAG 學習圖增強 LLM 推理的 pattern，但不引入 GraphRAG SDK / 框架（太重、依賴鏈複雜）。自行在 networkx + pmi-core 內實作核心子集。 The GraphRAG pattern of graph-augmented LLM reasoning is adopted conceptually, but the GraphRAG SDK/framework itself is not — too heavy and complex. The relevant subset is implemented directly in networkx + pmi-core.

為什麼這樣做 Why — three key consequences

LLM 成本攤平

Loop A 的 edge-proposer LLM 呼叫全域執行一次，跨所有 index 共用結果。Scoring tick（Loop B）不再呼叫 LLM 做關係判定，只讀已存邊。

Edge classification LLM calls run globally once and are shared across all indexes. Scoring ticks no longer call LLM for structural judgment — read-only against stored edges.

結構邏輯可擴充

collapse / partition / flip 從寫死的 code hacks 搬到圖結構 + formula registry。新增互斥 family 或翻轉邏輯只需加邊、加 formula，不需改 engine core。

Collapse / partition / flip move from hard-coded hacks to graph structure + formula registry. New mutual-exclusion families or flip logic require only new edges and a registered formula — no engine-core changes.

完整 reproducibility

ts_index_scores 帶 edge lineage：任何一個歷史分數都能還原當時用了哪些邊、哪個 formula、哪些 evaluation。接 §9 Enterprise audit 賣點。

ts_index_scores carries edge lineage: any historical score can be fully reconstructed from its edges, formula, and component evaluations. Directly supports the Enterprise audit-log selling point from §9.
